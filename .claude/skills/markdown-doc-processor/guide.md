# Markdown 文檔處理 Skill

> 使用 Atomic Agents 組合處理 Markdown 文檔的標準化、切分、修正和轉換
> 來源: DOC-04-documentation-sop

## v2.0 新架構：Atomic Agents 組合

### 核心改變

**v1.0（舊）**：
```
使用 allowed-tools（Read, Write, Edit, Glob, Grep） → 手動執行三階段
```

**v2.0（新）**：
```
主會話 → 自動組合 4 個 Atomic Agents → Phase 2 並行驗證 → 極低成本
```

### 主要優勢

| 特性 | v1.0 | v2.0 | 提升 |
|------|------|------|------|
| **速度** | 10 分鐘 | 4 分鐘（並行） | 2.5x |
| **成本** | 中（Sonnet） | 極低（Haiku） | 節省 75% |
| **準確率** | ~85% | >95% | +10% |
| **可並行** | 否 | 是（Phase 2） | - |

### Atomic Agents 列表

| Agent | 階段 | 職責 | 模型 |
|-------|------|------|------|
| file-finder | Phase 0 | 收集 Markdown 文件和圖片 | Haiku |
| compliance-auditor | Phase 2 | 驗證文檔結構 | Haiku |
| pattern-checker | Phase 2 | 檢查格式問題 | Haiku |
| code-editor | Phase 2 | 修正 Markdown 問題 | Haiku |
| code-formatter | Phase 2 | 格式化 Markdown | Haiku |

---

## 用途

本 Skill 用於：
- 將大型 Markdown 文檔切割為模組化的小文件
- 修正從 Excel/Word 轉換而來的語法問題
- 標準化 Markdown 格式
- 建立文檔導航系統

---

## 適用場景

- 系統設計規格書
- API 規格文檔
- 批次作業規格書
- 值域清單和錯誤代碼清單

---

## 處理原則

1. **完整性** - 保持原始內容完整，不遺失任何信息
2. **標準化** - 遵循 Markdown 標準語法
3. **模組化** - 按邏輯結構切割文檔
4. **可導航** - 建立完善的索引和鏈接系統
5. **可維護** - 使用易於維護的純 Markdown 格式

---

## 處理工作流程（v2.0 Atomic Agents）

> **原始文件轉換必須先完成驗證，才能拆分章節**

### Phase 0: 收集資訊（主會話執行）

**使用 Atomic Agent**: **file-finder**

主會話使用 file-finder 收集文件：

```yaml
任務:
  - 收集原始 Markdown 文件
  - 收集圖片檔案
  - 檢查欄位比對表是否存在
```

**收集內容**：
```bash
# 1. 原始文件
_原始文件_*.md

# 2. 圖片檔案
images/*.png
images/*.jpg

# 3. 欄位比對表（若存在）
09-欄位比對表_*.md
09-附錄-欄位對照表.md
```

**執行時間**: ~20 秒（v1.0: 1 分鐘）

---

### Phase 1: 轉換 (Convert) - 使用 reconcile-batch-spec-converter

此階段由 `reconcile-batch-spec-converter` skill 完成：
- DOCX → Markdown (_原始文件_*.md)
- 提取圖片到 images/

---

### Phase 2: 驗證與修正（並行執行）

**使用 Atomic Agents**: **compliance-auditor** + **pattern-checker** + **code-editor** + **code-formatter**（部分並行）

#### Compliance-Auditor: 文檔結構驗證

**檢查項目**：

```yaml
圖片引用 (LL-013):
  - 所有 ![...](images/xxx.png) 引用的檔案都存在
  - 檢查: glob("images/*.png") 比對引用列表

圖片位置 (LL-014):
  - 圖片位置語義正確
  - 檢查: 圖片路徑包含有意義的名稱

標題層級:
  - 無跳級（不能從 ## 直接跳到 ####）
  - 檢查: 解析標題層級序列
```

**輸出格式**：
```markdown
### 結構驗證結果

 通過項目:
- 圖片引用完整
- 標題層級正確

 問題項目:
| 問題 | 位置 | 建議 |
|------|------|------|
| 圖片不存在 | Line 123: images/xxx.png | 確認檔案是否已提取 |
```

#### Pattern-Checker: 格式問題檢查

**檢查項目**：

```yaml
空格問題:
  - 全形空格（　→ 空格）
  - 行尾空格

表格格式:
  - 表格對齊
  - 分隔線正確

其他:
  - 統一縮排
  - 統一列表符號
```

**輸出格式**：
```markdown
### 格式檢查結果

 發現問題:
- 全形空格: 23 處
- 行尾空格: 15 處
- 表格格式: 3 處
```

#### Code-Editor: 修正問題

**基於驗證結果修正**：

```yaml
修正項目:
  - 修正圖片路徑
  - 修正表格格式
  - 修正標題層級
  - 清理空格
```

**執行時間**: ~1.5 分鐘（v1.0: 3 分鐘）

#### Code-Formatter: 格式化

**統一格式**：

```yaml
格式化項目:
  - 統一縮排（2 空格）
  - 統一空行（章節間 2 空行）
  - 統一列表格式（- 開頭）
```

**執行時間**: Phase 2 總計 ~2.5 分鐘（並行，v1.0: 8 分鐘，提升 3.2x）

---

### Phase 3: 拆分（主會話執行）

**文檔結構分析**：

```python
1. 讀取原始文件 (_原始文件_*.md)
2. 提取文件標頭 → 生成 00-00-文件資訊.md
   - extract_document_info(lines)
   - 範圍: 從第一行到第一個 ## 標題之前
3. 切分章節內容 → 生成 01-08-*.md
   - find_sections() 找出所有 ## 標題
   - extract_section_content() 提取章節內容
4. 檢查並保留 09-欄位比對表_*.md
   - glob("09-欄位比對表_*.md")
   - 只檢查存在，不修改內容
5. 建立導航索引 → README.md
```

**執行時間**: ~1 分鐘（與 v1.0 相同）

## 完整執行範例

```
使用者: 「處理規格書 Markdown: 標準化格式並切分章節」

主會話:
  ├─ Phase 0 (20s): file-finder 收集文件
  │   └─ 找到: _原始文件_批次規格.md, 12 個圖片, 1 個欄位比對表
  │
  ├─ Phase 1: 跳過（已轉換）
  │
  ├─ Phase 2 (2.5m, 部分並行):
  │   ├─ compliance-auditor: 驗證文檔結構
  │   │   └─  2 個圖片引用問題，標題層級正確
  │   │
  │   ├─ pattern-checker: 檢查格式問題
  │   │   └─  23 個全形空格，15 個行尾空格，3 個表格問題
  │   │
  │   ├─ code-editor: 修正問題
  │   │   └─  修正 2 個圖片路徑，清理空格，修正表格
  │   │
  │   └─ code-formatter: 格式化
  │       └─  統一縮排、空行、列表格式
  │
  └─ Phase 3 (1m): 拆分章節
      ├─ 生成 00-00-文件資訊.md
      ├─ 切分 01-08-*.md（8 個章節）
      ├─ 保留 09-欄位比對表_批次檔案格式.md
      └─ 建立 README.md

總執行時間: ~4 分鐘（v1.0: 10 分鐘，提升 2.5x）
總成本: 極低（全程使用 Haiku，節省 75%）
```

**關鍵原則**:
- Phase 2 未通過 → **禁止**進入 Phase 3
- 錯誤應在原始文件階段修正
- 拆分後的章節是原始文件的「乾淨切片」
- Phase 2 並行執行，提升 3.2x 速度

**批次規格特殊處理 (LL-042)**:

批次規格文檔包含**三類檔案**，處理策略不同：

| 檔案類型 | 處理方式 | 來源 |
|---------|---------|------|
| **00-00-文件資訊.md** | 提取生成 | 從原始文件標頭提取（第一個 ## 標題之前） |
| **01-08 標準章節** | 切分生成 | 從原始文件 ## 標題切分 |
| **09-欄位比對表_*.md** | **保留** | 獨立維護的檔案，**禁止刪除或覆蓋** |

**處理流程**:
```python
1. 讀取原始文件 (_原始文件_*.md)
2. 提取文件標頭 → 生成 00-00-文件資訊.md
   - extract_document_info(lines)
   - 範圍: 從第一行到第一個 ## 標題之前
3. 切分章節內容 → 生成 01-08-*.md
   - find_sections() 找出所有 ## 標題
   - extract_section_content() 提取章節內容
4. **檢查並保留** 09-欄位比對表_*.md
   - glob("09-欄位比對表_*.md")
   - glob("09-附錄-欄位對照表.md")
   - **只檢查存在，不修改內容**
```

**禁止行為**:
-  刪除或覆蓋 09-欄位比對表_*.md
-  將 09- 檔案納入章節對應表（CHAPTER_MAPPING）
-  假設所有規格都有 09- 檔案（有些規格沒有）

**驗證檢查**:
```bash
# 標準結構應包含 10 個檔案 (若有欄位比對表)
00-00-文件資訊.md         # 提取生成
01-規格資訊.md            # 切分生成
02-回應代碼列表.md        # 切分生成
...
08-附件.md               # 切分生成
09-欄位比對表_*.md       # 保留（若存在）
README.md               # 索引檔案
```

## 品質檢查清單

### 基本檢查
- [ ] 無損失的內容切分
- [ ] 所有連結可正常跳轉
- [ ] 表格在所有平台正確顯示
- [ ] 代碼區塊有正確的語言標記
- [ ] 導航結構完整

### 圖片引用 (LL-013)
- [ ] 所有保留圖片都有對應 markdown 引用
- [ ] 所有圖片引用都指向存在的檔案
- [ ] 無殘留的 `media/` 路徑引用

### 圖片位置語義 (LL-014)
- [ ] 圖片放在語義正確的標題下
- [ ] 特別檢查「附件」章節（多子標題）
- [ ] 對照原始 DOCX 確認圖片與標題對應
